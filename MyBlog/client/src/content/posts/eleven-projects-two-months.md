# 十一个项目，两个月，一个开发者的自白

Git 日志能告诉你的只有时间戳和提交信息。它不会告诉你那些深夜盯着空白控制台发呆的瞬间，也不会记录你为一个 bug 辗转反侧三天的夜晚。这篇与其说是技术分享，不如说是对自己过去两个月的一个交代。

## 起点

这一切始于 2026 年 4 月 28 日。那天我提交了三个项目：一个番茄钟、一个待办事项应用、一个 Markdown 笔记应用。动机很朴素——想看看自己能不能在短时间内，从只会"写页面"变成真正能独立交付产品的人。

## 项目清单

按时间顺序排下来：

| 项目 | 时间 | 核心收获 |
|------|------|---------|
| 番茄钟 / Todo App / Markdown 笔记 | 第 1 天 | 基础状态管理和本地存储 |
| 天气查询应用 | 第 2 天 | 第三方 API 对接和错误处理 |
| AI 写作助手 | 第 3-4 天 | SSE 流式输出和 Prompt 工程 |
| AI 语音识别 | 第 5 天 | Web Audio API 和实时流式处理 |
| 在线商城 | 第 6-9 天 | 全栈 CRUD、安全加固、部署流程 |
| MyBlog + Hermes Agent 实践 | 第 10-14 天 | 前端架构、CI/CD、静态站点生成 |
| AI Chatbot RAG | 第 15-18 天 | LangChain 集成、文档检索、知识库 |
| 微信小程序 AI Agent | 第 19-20 天 | 小程序生态、跨平台开发 |

## 那些 Git 日志不会告诉你的

### 1. 做减法比做加法难十倍

在线商城是我投入时间最多的项目（将近 4 天），也是我最终决定不部署上线的项目。代码写完了，CRUD 通了，安全加固也做了。但我问自己一个问题：我真的需要维护一个商城吗？答案是不。

这个道理我是在写到第 6 个项目时才想明白的。**做一个项目真正的成本不是写代码的那几天，而是未来每一次维护、每一次重构、每一次读者带着问题来找你时，你必须有答案。**

### 2. 工具链能帮你，也能毁了你

Vite 8 + TypeScript 6 + React 19 的组合很香，但"太新"本身就是一个坑。SPA 部署到 GitHub Pages 时，Node 版本不够新导致的 `CustomEvent is not defined`，我花了整整半天才定位到根因。不是技术难，是问题被层层叠叠的新框架包裹着，很难一眼看穿。

学会了一件事：**遇到奇怪的错误，先去检查版本兼容性，而不是钻进源码里。** 这个原则后来帮我省了无数时间。

### 3. 流式输出是 AI 应用的门槛体验

AI 写作助手和 Chatbot 两个项目让我深刻理解了 SSE 在用户体验中的分量。一个不带流式输出的 AI 对话框，用户体验和带流式的完全是两个世界。前者让人焦躁，后者让人感觉"AI 在思考"。

实现本身并不复杂——一个 `ReadableStream` + `TextDecoder` 循环读取就够了。但边界情况很多：网络中断怎么恢复、流式内容太长怎么截断、刷新页面后历史怎么保留。**做 demo 很容易，做产品很难。**

### 4. RAG 不只是一种技术，是一种架构思维

Chatbot 项目让我第一次真正理解 RAG。之前觉得 RAG 就是把文档切碎、向量化、存起来，用户问问题时去搜一下就行。

实际实现后我发现，真正的 RAG 工程不在向量检索本身，而在：

- 文档怎么切才能让每一块都有独立语义
- Prompt 怎么写才能让模型理解"你有知识库，但不知道也不要瞎编"
- 引用来源怎么做才能让用户信任而不是疑惑

Git 日志里这个项目只有几次提交，但实际上我调整了 6 版 Prompt 才让回答质量达到自己满意的水平。

### 5. 部署是最后一面墙

从 4 月 30 日到 5 月 16 日，关于部署的提交贯穿了整个项目周期。GitHub Actions 配置、Vercel 兼容、base 路径双平台适配、SPA 路由 404 处理——每一个单独拎出来都不难，但它们像多米诺骨牌一样倒着来。

5 月 16 日那次"博客搬家"让我损失了 13 个文件和 7 篇已发布的文章。虽然最终全部恢复了，但那次事故让我养成了一个习惯：**在 git clean 之前，永远 `git stash` 或者建个分支。** 这个习惯后来救过我两次。

## 如果让我重新来

如果再给我两个月，我会做几件不同的事：

1. **少做两个项目，多做一次深度重构。** 11 个项目意味着每个都是浅尝辄止，不如把博客或商城中的一个做到 90 分。

2. **从一开始就写测试。** 到目前为止，这些项目几乎没有单元测试。当时觉得"先做出功能再说"，但功能做完了，回头补测试的动力就没了。如果让我排优先级，我会在每个项目的第一批提交里就包含测试脚手架。

3. **写更多的文档。** 有些代码现在回头看，连我自己都要想一会儿才能明白当时为什么这么写。README 写了，但架构决策记录（ADR）一个都没有。

4. **早点使用 monorepo 工具。** 项目多起来后，依赖管理变得很混乱。每个子项目都要 `npm install` 一遍，版本不一致的问题反复出现。

## 最意外的收获

管理这个项目的过程中，Git 变成了我最重要的工具——不仅是版本控制，更是项目管理的日志。通过 `git log --oneline --graph` 看到 60 多次提交组成的项目演进轨迹，比任何甘特图都直观。

另一个意外收获是**发布本身带来的正反馈**。把博客部署到公网后，每篇文章都有真实的访客（不多，但存在），这比任何"项目完成"的自我感觉都更能推动人继续前进。

## 下一步

博客会继续写，Chatbot 会继续迭代。但接下来两个月，我的目标不是"做更多"，而是"做得更好"：

- 给博客加上全文搜索
- 把 Chatbot RAG 嵌入博客页面
- 选一个项目做完整的测试覆盖
- 为这个 monorepo 整理一份清晰的开发指南

两个月前我觉得"学完全栈"是一条有终点的路。现在我知道了——**编程不是爬一座山，是走进一片森林。你永远看不到边界，但每多走一步，眼前都会出现新的风景。**

*2026 年 5 月 17 日*
